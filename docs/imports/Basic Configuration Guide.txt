I. CẤU HÌNH DNS
Công dụng:
DNS cho phép truy cập các dịch vụ web bằng tên miền dễ nhớ thay vì địa chỉ IP.
Dịch vụ ứng dụng Cloudflare đều hoạt động thông qua proxy ngược (reverse proxy), được bật trên bản ghi DNS (DNS record).
Bước 1: Mở zone để cấu hình
Bước 2: Thêm một bản ghi DNS 
Tại thanh menu bên trái dashboard, chọn DSN > Records > chọn Add record
Tạo một bản ghi A mới trỏ về zone root. Đảm bảo luôn giữ nút chuyển đổi Proxy BẬT để Cloudflare thực hiện proxy ngược trên miền này. Điền các thông tin tương tự như sau (có thể thay đổi theo thông tin mà chúng ta cần thiết lập, vì đây là bài lab có sẵn):
Type: A
Name: @
IPv4 address: 20.88.188.200
Proxy: yes
Comment: Lab origin server
 
Bước 3: Kiểm tra bằng cách mở terminal và truy vấn đến bản ghi DNS
Có thể thấy rằng mặc dù địa chỉ của server gốc là 20.88.188.200 nhưng kết quả truy vấn trả về là hai địa chỉ IP của Cloudflare (104.18.4.167 và 104.18.5.167).
Chúng ta thử tắt nút Proxy và truy vấn lại bản ghi, xem chuyện gì sẽ diễn ra:
Địa chỉ của server gốc là 20.88.188.200 đã bị lộ, do không bật nút proxy
Điều này chứng minh rằng Cloudflare đang hoạt động như một reverse proxy cho tất cả các yêu cầu đến tên miền mới - nó chấm dứt HTTP session từ người dùng và sau đó thiết lập một phiên mới cho server gốc. 
Bây giờ, chúng ta thử sử dụng trình duyệt web để gõ tên bản ghi DNS vừa thiết lập, kết quả trang web sẽ hiển thị giao diện như hình bên dưới.
II. CẤU HÌNH TLS (SSL)
Công dụng: để mã hóa kết nối giữa người dùng, Cloudflare và server gốc.
Cloudflare tự động cấp Universal Certificate cho tất cả domain được thêm vào Cloudflare.
Universal Certificate bảo vệ apex domain và 1 cấp subdomain thông qua wildcard (ví dụ *.example.com - www.example.com - mail.example.com ).
Nếu hệ thống dùng subdomain nhiều cấp hơn, cần đăng ký Advanced Certificate.
Advanced Certificate cho phép tùy chỉnh nâng cao, như cấu hình cipher suites và các tham số TLS khác. 
Cipher suites: bộ thuật toán mã hóa được dùng trong quá trình TLS handshake ( trao đổi khóa, mã hóa dữ liệu, kiểm tra toàn vẹn)
Demo:
Trước tiên, hãy đảm bảo tắt mọi dịch vụ đang chạy trên máy có thể chặn lưu lượng HTTPS (VPN, Cloudflare WARP, v.v.).
Bước 1: 
Luôn luôn bật https (Always use https)
Điều này sẽ giúp ngăn chặn các yêu cầu vô tình qua HTTP không sử dụng TLS.
Chọn SSL/TLS > Edge Certificates > bật Always Use Https
Thay đổi phiên bản TLS tối thiểu (Change minimum TLS Version)
Thông số mặc định là TLS 1.0, tuy nhiên đây là phiên bản khá cũ nên cần thiết lập TLS 1.2 để đáp ứng các tiêu chuẩn bảo mật hiện nay như PCI, DSS, tài chính, ngân hàng)
PCI (payment card industry), DSS (data security standard) tiêu chuẩn bảo mật bắt buộc cho mọi hệ thống lưu trữ, xử lý hoặc truyền dữ liệu thẻ thanh toán

Như kỳ vọng, giao diện của trang web vẫn hiện thị, và nó cũng được bảo vệ bởi universal certificate của Cloudflare.
Ngoài ra còn có thể sử dụng openssl để truy suất và thấy rõ rằng Universal certificate từ Cloudflare tự động cấp cho miền apex và các subdomain trực tiếp của nó

Điền sub2.sub1.precise-voyager.sxplab.com  vào mục Certificate Hostnames

IV. CẤU HÌNH BOT MANAGEMENT
Công dụng: 
Ngăn chặn các cuộc tấn công tự động làm tiêu tốn tài nguyên và doanh thu, bao gồm credential stuffing, scalping và content scraping.
Gán điểm bot (bot score) cho mỗi request dựa trên machine learning, huấn luyện trên hơn 25 triệu website, giúp phân biệt chính xác bot hợp lệ và bot độc hại.
Triển khai tức thì thông qua WAF rule template như Define Bots và Likely Bots — chỉ cần bật rule và chọn hành động phản hồi.
Bật tính năng  JavaScript Detection
Tham khảo tại: JavaScript Detections · Cloudflare challenges docs 
Phát hiện JavaScript (JavaScript Detection) là một cơ chế xác thực độc lập, tách biệt với Challenge Pages và Turnstile của Cloudflare. Tính năng này hỗ trợ các giải pháp Bot Management của Cloudflare trong việc nhận diện và phân loại các yêu cầu truy cập tự động (automated requests).
Khác với Challenge Pages hoặc Turnstile, vốn chỉ thu thập tín hiệu từ phía trình duyệt tại thời điểm xác thực, JavaScript Detection hoạt động liên tục trên mọi yêu cầu gửi đến website. Cơ chế này thực thi mã JavaScript phía máy khách để thu thập thêm các tín hiệu hành vi và môi trường truy cập, từ đó tăng độ chính xác trong việc xác định lưu lượng hợp lệ và phát hiện bot.
Để bật tính năng này, tại giao diện dashboard, chọn Security > Settings > Bot Traffic > Biểu tượng chỉnh sửa (JS Detections) > Bật Javascript Detections

Dưới đây là một số tình huống cấu hình phổ biến liên quan đến bot để tham khảo:
3.1 Allow traffic from search engine bots
Ví dụ về quy tắc này sẽ áp dụng cơ chế thử thách (challenge) đối với các yêu cầu truy cập đến từ danh sách quốc gia được chỉ định, đồng thời vẫn cho phép lưu lượng từ các bot công cụ tìm kiếm như Googlebot, Bingbot và các bot đã được xác minh khác (theo danh sách từ cloudflare) tiếp tục truy cập.
Bước 01: Tại dashboard chọn mục Security > Security rules > Create rule > Custom rules
Bước 02: Nhập tên rule hiện tại và điền đầy đủ các thông tin theo hình bên dưới. Sau đó chọn vào nút Deploy để triển khai rule đã cấu hình.
3.2 Bảo vệ chống lại các bot độc hại
Để giảm thiểu và ngăn chặn lưu lượng bot độc hại, cần cấu hình tuần tự 03 rule như sau:
Rule 01: Skip verified bots
Bước 01: Tại dashboard chọn mục Security > Security rules > Create rule > Custom rules
Bước 02:  Nhập tên rule hiện tại và điền đầy đủ các thông tin theo hình bên dưới. Sau đó chọn vào nút Deploy để triển khai rule đã cấu hình.
Khi chọn All remaining custom rules có nghĩa là cho phép bỏ qua toàn bộ các Custom Rules còn lại đối với những bot đã được Cloudflare xác minh là hợp lệ. Điều này giúp tránh việc các bot tin cậy bị áp dụng nhầm cơ chế kiểm soát hoặc challenge.
Rule 02: Block definitely automated
Bước 01: Tại dashboard chọn mục Security > Security rules > Create rule > Custom rules
Bước 02:  Nhập tên rule hiện tại và điền đầy đủ các thông tin theo hình bên dưới. Sau đó chọn vào nút Deploy để triển khai rule đã cấu hình.
Mục đích của rule này là sẽ chặn toàn bộ các request có Bot Score = 1.
Trong đó, Bot Score là điểm đánh giá của Cloudflare để xác định mức độ một request có phải bot hay không (thang điểm tính từ 1–99):
Automated (bot score = 1): Cloudflare đánh giá gần như chắc chắn là lưu lượng tự động.
Likely automated (bot scores 2 - 29): khả năng cao là bot tự động hoặc độc hại.
Likely human (bot scores 30-99): được đánh giá khả năng cao là người dùng thật.
Dựa vào cách đánh giá trên nên mức độ tin cậy rất cao, do đó việc áp dụng Action = Block thường có rủi ro chặn nhầm (false positive) không đáng kể, đồng thời giúp giảm lưu lượng bot độc hại trước khi vào hệ thống phía sau.
Rule 03: Challenge likely automated
Bước 01: Tại dashboard chọn mục Security > Security rules > Create rule > Custom rules
Bước 02:  Nhập tên rule hiện tại và điền đầy đủ các thông tin theo hình bên dưới. Sau đó chọn vào nút Deploy để triển khai rule đã cấu hình.
Ở rule này, khi thiết lập các thông số như hình bên trên thì Cloudflare sẽ áp dụng Managed Challenge đối với các request có Bot Score từ 2 đến 29. Khoảng điểm này thể hiện lưu lượng có dấu hiệu giống hành vi tự động (likely automated behavior), tuy nhiên chưa đủ chắc chắn để chặn hoàn toàn. Do đó, với Managed Challenge, Cloudflare sẽ tự động áp dụng cơ chế xác thực phù hợp (có thể không hiện CAPTCHA nếu không cần thiết) để cho phép người dùng hợp lệ tiếp tục truy cập và ngăn chặn bot hoặc truy cập tự động.
Cách xử lý này giúp cân bằng giữa bảo mật và trải nghiệm người dùng, đồng thời giảm rủi ro chặn nhầm so với việc block trực tiếp.
3.3 Bảo vệ chống lại Scrapping

V. CẤU HÌNH CACHE TUNING
Công dụng: 
Tăng tốc phân phối nội dung bằng cách phục vụ dữ liệu từ 300+ edge toàn cầu, giúp giảm tải cho origin, giảm độ trễ và tiết kiệm băng thông
Kiểm soát hành vi cache rất chi tiết thông qua Cache Rules (TTL, cache key, cách xử lý response từ origin) mà không cần chỉnh code
Vẫn phục vụ nội dung cũ (stale) khi origin gặp lỗi, nhờ cơ chế grace period, giúp hệ thống luôn sẵn sàng ngay cả khi backend có sự cố

Khi nào một nội dung được cache?
Một resource được Cloudflare cache khi đủ các điều kiện sau (ít nhất về mặt logic):
① Nội dung static / ổn định
JS, CSS, image, font
HTML giống nhau cho mọi user
② Origin cho phép cache
Có header hợp lệ, ví dụ:
Cache-Control: public
Cache-Control: max-age=...
Không có private, no-store, no-cache
③ Method phù hợp
GET hoặc HEAD
Không cache POST (mặc định)
④ Có TTL rõ ràng
Origin trả TTL
hoặc
Cloudflare Cache Rule gán TTL
⑤ Không phụ thuộc dữ liệu cá nhân
Không cần cookie để render nội dung
Không dựa vào header riêng của user (Authorization, JWT…)
2. Create a cache rule for /services/luggages
In the Cloudflare Dashboard go to Caching ‣ Cache Rules and select Create rule
Enter a descriptive name for the rule, such as “Cache HTML on Luggages Page”
Set the Field to URI Path
Set the Operator to equals
Set the Value to /services/luggages/
Set Cache status to Eligible for cache
You can customize additional cache settings such as:
Edge TTL: → Thời gian Cloudflare lưu nội dung trên edge. TTL càng dài thì cache hit càng cao, origin càng nhẹ.
Browser TTL:  → Thời gian trình duyệt của người dùng lưu cache. Giúp user load lại nhanh hơn, giảm request quay lại Cloudflare.
Cache Key:  → Quyết định Cloudflare coi hai request có giống nhau hay không (URL, query string, header…). Cache key đúng giúp tránh cache sai nội dung.
Serve Stale Content:  → Khi cache hết hạn, Cloudflare vẫn tạm trả nội dung cũ trong lúc âm thầm lấy bản mới từ origin → không gián đoạn trải nghiệm.
Respect Strong ETags:  → Cloudflare dùng ETag mạnh từ origin để kiểm tra nội dung có thay đổi hay chưa → tiết kiệm bandwidth mà vẫn đúng dữ liệu.
Origin Error Page Pass-thru: → Khi origin lỗi, Cloudflare trả nguyên trang lỗi từ origin thay vì trang lỗi mặc định của Cloudflare.
To the right of where you see Edge TTL, Click + Add Setting
Choose Ignore cache-control header and use this TTL and set the duration to 30 seconds
